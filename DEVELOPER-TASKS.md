## Primary developer tasks (what to do first)

If you only have 30-60 minutes, do these in order:

1. Run focused Atari/Genesis validation pack and confirm all exit codes are zero.
2. Do one manual testing pass for the feature you changed and record notes.
3. Add or update one benchmark/test around the code you touched.
4. Capture screenshots for any user-visible behavior changes.
5. Update issue status with concrete evidence (commands + outputs + commit).

## Atari 2600 and Genesis test support checklist

This tells you exactly what to create and maintain.

### Local reference ROM folders (outside repo)

Create local ROM folders (do not commit ROMs):

- `C:\~reference-roms\atari2600\`
- `C:\~reference-roms\genesis\`

Suggested naming examples:

- `C:\~reference-roms\atari2600\example-atari-title-a26.bin`
- `C:\~reference-roms\genesis\example-sonic.bin`

### Repository files and folders to use

- Example regression manifest scaffold: [~docs/testing/reference-rom-validation-manifest.example.json](~docs/testing/reference-rom-validation-manifest.example.json)
- Artifact output folder (generated by script): [artifacts/reference-validation](artifacts/reference-validation)
- Automation script: [scripts/reference-validation-pack.ps1](scripts/reference-validation-pack.ps1)
- Parity command guide: [~docs/plans/platform-parity-benchmark-and-correctness-gates.md](~docs/plans/platform-parity-benchmark-and-correctness-gates.md)

### Minimum command set

Run this first:

```powershell
.\scripts\reference-validation-pack.ps1 -SkipBuild
powershell -File scripts/compare-reference-validation.ps1 -BaselineDir artifacts/reference-validation-baseline -CandidateDir artifacts/reference-validation
powershell -File scripts/test-reference-validation-compare.ps1
```

Expected generated outputs:

- `artifacts/reference-validation/atari-harness.txt`
- `artifacts/reference-validation/genesis-harness.txt`
- `artifacts/reference-validation/atari-genesis-benchmarks.json`
- `artifacts/reference-validation/reference-validation-summary.json`
- Optional drift report: `artifacts/reference-validation/drift-report.json`

### How you can directly help parity work

1. Replace placeholder SHA256 values in the example manifest with real values from your local ROM set.
2. Capture first known-good digest outputs from harness runs and fill the digest placeholders.
3. Attach failing harness outputs to issues when regressions appear.
4. Keep a small stable ROM sample set so comparisons stay consistent across sessions.

## Screenshot and docs update plan

When behavior/UI changes, follow this plan:

1. Capture screenshots during manual testing.
2. Save images under [docs/screenshots](docs/screenshots) in the matching feature folder.
3. Use and update checklist: [docs/screenshots/CAPTURE-CHECKLIST.md](docs/screenshots/CAPTURE-CHECKLIST.md)
4. Update the relevant user docs page:
   - Save states: [docs/Save-States.md](docs/Save-States.md)
   - Rewind: [docs/Rewind.md](docs/Rewind.md)
   - Movie system: [docs/Movie-System.md](docs/Movie-System.md)
   - TAS editor: [docs/TAS-Editor-Manual.md](docs/TAS-Editor-Manual.md)
   - Debugging: [docs/Debugging.md](docs/Debugging.md)
   - Video/audio: [docs/Video-Audio.md](docs/Video-Audio.md)
5. Add one short changelog-style note in your issue comment that references updated docs/screenshots.

## Icon approval workflow

If icon or visual assets change:

1. Capture before/after screenshots.
2. Record where icon files live and where they are used.
3. Request explicit approval in the issue with side-by-side images.
4. After approval, update docs screenshots where needed.
