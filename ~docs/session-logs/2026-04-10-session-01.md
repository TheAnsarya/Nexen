## Work Done

### Comprehensive Hot-Path Audit

Ran a full audit of 30+ source files across all emulated systems. Identified 10 findings:

- **HIGH**: SNES `IMemoryHandler` virtual dispatch on every memory Read/Write (~3.58M virtual calls/sec)
- **MEDIUM**: SPC700 double-precision multiply, VS Dual per-frame 240KB allocation, GetPortStates vector rebuild, NES PPU config chain
- **LOW**: Various minor items (SPC ExitExecLoop, debugger UI dispatch, NotificationManager)
- **Confirmed clean**: GBA memory manager (switch-based), NES ReadRamFast, SnesPpu scanline, SoundMixer, LogDebug, CheatManager

### SPC Fixed-Point Analysis (Rejected)

Investigated converting SPC `_clockRatio` double multiply to fixed-point. Rejected due to:

- Master clock overflows uint64 at large values (770 billion after 10 hours)
- Affects audio timing accuracy (critical for game compatibility)
- `_clockRatio` serialized to save states — compatibility concern
- Benefit only ~5-10µs/frame — not worth the risk

### SNES Memory Handler Devirtualization (Implemented)

Added direct-access pointer fields (`_directReadPtr`, `_directWritePtr`, `_directMask`) to `IMemoryHandler`. RAM/ROM handlers set these in constructors, enabling `SnesMemoryManager::Read()`/`Write()`/`ReadDma()`/`WriteDma()` to bypass virtual dispatch with an inline array access for the common case.

- `RamHandler`: Sets both read and write direct pointers
- `RomHandler`: Clears `_directWritePtr` (read-only)
- `BsxMemoryPackHandler`: Clears both pointers (flash protocol needs custom Read/Write)
- All other handlers (register, coprocessor) default to nullptr — remain on virtual path

### VS Dual Per-Frame Allocation Fix (Implemented)

Moved `make_unique<uint16_t[]>(240KB)` from per-frame allocation to lazy member in `BaseNesPpu::_vsDualMergedBuffer`. Only affects VS Dual System "Both" video output mode.
