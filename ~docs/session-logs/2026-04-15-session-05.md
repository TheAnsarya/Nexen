## Summary

Implemented issue #1295: automatic fallback/migration when portable config path is not writable.

This hardens startup path selection so settings persistence remains functional even when the executable folder has a read-only or restricted permissions profile.

## What Changed

In `UI/Config/ConfigManager.cs`:

1. Added `IsFolderWritable(string folder)` probe using temp-file create/delete.
2. Added `TryMigrateSettingsFile(string sourceConfigFile, string destinationConfigFile)` helper.
3. Added `ResolveHomeFolderPath(...)` decision logic:
	- Use portable if portable settings exists and folder is writable.
	- Use documents when portable settings exists but portable is not writable and documents is writable.
	- Migrate `settings.json` from portable to documents when needed.
	- Fall back to portable if both paths are non-writable.
4. Updated `HomeFolder` getter to use resolver and guarded directory-creation fallback.

In `Tests/Config/ConfigManagerHomeFolderTests.cs`:

- Added targeted unit tests for home-folder resolution and migration behavior.
