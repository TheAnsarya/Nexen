# Session 24 - Menu Crash Follow-up (Issue #1245)

## Summary

Followed up on user validation after Event Viewer recovery.

- Confirmed Event Viewer now opens successfully.
- Fixed remaining TAS menu crashes (`Tools -> TAS Editor -> Open`, `Create New Movie`, `Record`) by hardening TAS window control initialization.
- Kept Event Viewer initialization defensive to avoid null control dereferences.
- Updated release-facing save-state documentation and changelog entries to reflect shipped behavior changes.

## Root Cause

`TasEditorWindow` relied on generated control fields (`FrameList`, `PianoRoll`, `SearchBox`) during initialization and event hookup. In runtime cases these fields could be null, causing `NullReferenceException` during `InitializeComponent()` path and all TAS menu actions to fail.

## Fixes Implemented

- `UI/Windows/TasEditorWindow.axaml`
  - Converted key controls to `x:Name` for reliable named lookup.
- `UI/Windows/TasEditorWindow.axaml.cs`
  - Added resilient `FindControl` fallback binding for required controls.
  - Switched all direct uses to resolved private fields (`_frameList`, `_pianoRoll`, `_searchBox`).
  - Added explicit initialization exceptions with actionable messages if required controls are missing.
- `UI/Debugger/Windows/EventViewerWindow.axaml`
  - Updated `lstEvents` to `x:Name`.
- `UI/Debugger/Windows/EventViewerWindow.axaml.cs`
  - Added fallback control lookup (`FindControl`) for `picViewer` and `lstEvents` before ViewModel creation.
- Documentation updates for release prep:
  - `README.md`
  - `docs/Save-States.md`
  - `CHANGELOG.md`
  - `Core/Shared/SaveStateManager.h`
  - `UI/ViewModels/MainMenuViewModel.cs`
