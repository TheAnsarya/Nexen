# Session Log — 2026-04-08 Session 04

## Summary

Fixed Recent Play auto-save timing bug where the first save fired ~100ms after ROM load instead of waiting 5 minutes, producing a tiny/useless save state file.

## Issue

- **#1187** — Recent Play save fires immediately on ROM load (saves empty/minimal state)

## Root Cause

Two bugs:

1. **`_lastRecentPlayTime = 0` (Unix epoch)** in `SaveStateManager` constructor — `ShouldSaveRecentPlay()` checks `(now - _lastRecentPlayTime) >= 300` which is always true since `now - 0` is always ≥ 300 seconds
2. **`ResetRecentPlayTimer()` never called** — the method existed and was fully exported through interop (C++ → DLL → C#) but had zero call sites

### Timeline of Bug

1. ROM loads → `SaveStateManager` constructed with `_lastRecentPlayTime = 0`
2. UI timer ticks ~100ms later
3. `ShouldSaveRecentPlay()` returns true (`now − epoch ≥ 300`)
4. `SaveRecentPlayState()` saves to `recent_01.nexen-save`
5. State data minimal (few frames executed) → 3391 bytes vs ~11K for normal saves

## Fix

- **`SaveStateManager.cpp` constructor**: Changed `_lastRecentPlayTime = 0` → `_lastRecentPlayTime = std::time(nullptr)` (defense in depth)
- **`MainWindow.axaml.cs` GameLoaded handler**: Added `EmuApi.ResetRecentPlayTimer()` call so the 5-minute countdown starts fresh after ROM load

## Files Changed

- `Core/Shared/SaveStateManager.cpp` — Constructor initialization fix
- `UI/Windows/MainWindow.axaml.cs` — Added `ResetRecentPlayTimer()` call in `GameLoaded` handler
