# Session Log ŌĆö 2025-04-10 Session 01

## Summary

Investigated and fixed SNES performance regression caused by CDL recording overhead and rendering pipeline inefficiency.

## Problem

User reported "major performance problems when testing SNES emulation ŌĆö emulated very slow." Investigation needed to determine if Pansy/CDL logging was the cause, and if so, optimize it while keeping Pansy support.

## Root Cause Analysis

### Primary: CDL Virtual Call Chain

The `LightweightCdlRecorder::RecordRead()` method was called 2-4 times per instruction (~3-4M calls/sec on SNES). Each call triggered:

1. Virtual call to `IConsole::GetAbsoluteAddress()` (~5ns)
2. Switch statement in `SnesConsole::GetAbsoluteAddress()`
3. Handler table lookup in `MemoryMappings::GetAbsoluteAddress()`
4. Virtual call to `IMemoryHandler::GetAbsoluteAddress()` (~5ns)

Total: ~10-15ns ├Ś 3-4M reads/sec = ~30-60ms/sec of pure CDL overhead (2-4ms/frame at 60fps).

### Secondary: Rendering Pipeline Lock Contention

`SimpleImageViewer::DrawOperation` used 3 separate bitmap lock/unlock cycles per frame:

- Lock #1: Constructor ŌĆö `WriteableBitmap.Lock()` for `InstallPixels`
- Lock #2: `Render()` ŌĆö `DynamicBitmap.Lock(true)` for `DrawBitmap`
- Lock #3: `UpdateSurface()` ŌĆö `DynamicBitmap.Lock()` for frame copy

Additionally, `DispatcherPriority.MaxValue` was used for every frame's `InvalidateVisual`, potentially starving other UI operations.

### Confirmed NOT the Bottleneck

- **BackgroundPansyExporter**: Timer-based (configurable minutes interval), runs on threadpool ŌĆö not per-frame
- **SNES CPU execution**: Benchmarks showed 3-10ns per operation ŌĆö well optimized
- **CDL buffer operations**: Statistics, flag read/write, buffer copy all fast

## Changes Made

### CDL Page Cache (`LightweightCdlRecorder.h/.cpp`)

- Added `CdlPageEntry` struct with flat page cache (4KB pages)
- `BuildPageCache()` resolves every page at CDL start using virtual calls (one-time cost)
- Linearity verification ensures each cached page maps linearly to ROM
- `RecordRead()` now uses direct array lookup ŌĆö **0 virtual calls** (was 2)
- `RecordInstruction()` unchanged ŌĆö still uses virtual call for PC (1 per instruction)
- `RebuildPageCache()` method for consoles with dynamic mapping (NES bank switching)
- Null console guard for test compatibility
- `GetAddressSpaceSize()` helper derives cache size from `CpuType`

### Rendering Pipeline (`SimpleImageViewer.cs`, `SoftwareRendererView.axaml.cs`)

- Consolidated `DrawOperation` from 2 bitmap locks to 1 (InstallPixels + DrawBitmap in same lock scope)
- Eliminated potential stale memory issue (InstallPixels was reading buffer after lock release)
- Lowered `DispatcherPriority` from `MaxValue` to `Render` for frame invalidation

## Performance Impact

| Metric | Before | After |
|--------|--------|-------|
| RecordRead virtual calls | 2 per read | 0 per read |
| RecordInstruction virtual calls | 2 per instruction | 2 per instruction |
| Bitmap locks per frame | 3 | 2 |
| DispatcherPriority | MaxValue | Render |
| Estimated CDL overhead | ~10-15ns/read | ~3-5ns/read |
