# Session Log — 2026-04-14 Session 02

## Summary

Fixed WonderSwan PPU VTOTAL (port $16) frame boundary handling to match ares emulator behavior. This addresses issue #1076 (WSC FF2 fire magic video instability) which persisted through 4 previous fix attempts.

## Root Cause

Cross-emulator comparison between Nexen and ares revealed three critical behavioral differences when VTOTAL < 144:

1. **Frame duration**: ares always runs to `max(144, vtotal+1)` scanlines. Nexen terminated frames at `LastScanline+1`, creating short frames with disrupted CPU/game timing.
2. **Rendering Y wrapping**: ares uses `y = vcounter % (vtotal+1)` so the image repeats. Nexen had no wrapping — rendering stopped at the short frame boundary.
3. **VBlank inhibit**: ares inhibits VBlank when `vtotal < 144`. Nexen also inhibited VBlank but simultaneously disrupted frame timing by terminating early.

## Changes Made

### Core\WS\WsPpu.h

- Added `_prevRenderY` field — stores the rendering Y from the previous scanline's ProcessHblank for use by Exec's framebuffer output
- Modified `Exec()`:
	- Sprite copy now conditional: `LastScanline >= ScreenHeight` (no sprite copy for short VTOTAL)
	- Framebuffer output uses `_prevRenderY` instead of `Scanline - 1` for correct wrapping
- Updated template declarations for `DrawScanline`, `DrawSprites`, `DrawBackground` to accept `renderY` parameter

### Core\WS\WsPpu.cpp

- `ProcessHblank()`: Computes `renderY = Scanline % (LastScanline + 1)` and stores in `_prevRenderY`
- `ProcessEndOfScanline()`:
	- Frame boundary changed from `Scanline > LastScanline` to `Scanline > max(ScreenHeight, LastScanline)`
	- VBlank only fires when `LastScanline >= ScreenHeight` (144)
- `DrawScanline()`, `DrawSprites()`, `DrawBackground()`: Use `renderY` parameter for tile/sprite Y calculations instead of `_state.Scanline`
