## Technical Details

### Bug 1: Firmware Auto-Selection (#1182)

**Problem:** When loading a Lynx game, if the firmware file exists but has the wrong hash, Nexen shows a mismatch dialog even though it knows the expected hash. User must manually browse for the correct file.

**Fix:** Before showing any dialog, `FindMatchingFirmwareInFolder()` enumerates all files in the firmware folder, checks size match, then computes SHA-256 for size-matched files. If a file's hash matches any known hash for the firmware type, it's auto-copied to the expected filename. The dialog is only shown if no auto-match is found.

### Bug 2: Emulation Crash/Freeze (#1183)

**Problem:** `Emulator::Run()` had no exception handler around the main emulation loop. If emulation crashed (e.g., from bad firmware data), the emulation thread died silently while the UI remained in a "running" state — menus disabled, no error shown.

**Fix:** Wrapped the frame loop body in try/catch that handles `std::exception` and catch-all. On exception, the error is logged, an OSD message is displayed, and `_stopFlag` is set so the loop exits cleanly. The emulation thread returns normally, allowing the UI to process the `EmulationStopped` notification and return to an idle state.

**Note:** Windows SEH exceptions (access violations) are NOT caught by C++ try/catch under `/EHsc`. However, the Lynx emulator has proper bounds checking on all memory arrays, so the more likely failure mode with wrong firmware is garbage execution (caught by address out of range, bad opcode, etc.) rather than a raw access violation.
