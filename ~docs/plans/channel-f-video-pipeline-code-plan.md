# Channel F Video Pipeline — Code Plan (#977)

## Overview

The Channel F video system is unique: the F8 CPU directly writes to VRAM through I/O ports. There is no PPU chip — the video signal is generated by simple logic reading VRAM sequentially. This plan covers implementing a cycle-accurate video pipeline.

## Hardware Architecture

### VRAM Layout

- **8192 bytes** total (128 × 64 pixels × 2 bits/pixel internally, but stored as bytes)
- Organized as 64 rows × 128 columns
- Each byte stores one pixel's 2-bit color value (0-3)
- Colors: 0=Background (black), 1=Green, 2=Blue, 3=White/Gray

### I/O Ports

| Port | Direction | Purpose |
|------|-----------|---------|
| 0 | Out | Background color / misc control |
| 1 | Out | VRAM X position (0-131: 125 visible + overscan on each side) |
| 4 | In/Out | VRAM color data (2-bit, via port 5) |
| 5 | In/Out | Combined: X auto-increment + color + sound toggle |

### Write Sequence (from F8 CPU)

1. Set Y via port 0 bits (row select)
2. Set X via port 1 (column select, 0-131)
3. Write color via port 5 (which writes port 4 + auto-increments X)
4. Repeat step 3 for consecutive pixels

### Current Implementation (ChannelFMemoryManager)

The current code handles VRAM writes in `WritePort()`:
- Port writes update `_videoColor`, `_videoX`, `_videoY`
- VRAM is stored as `_vram[8192]`
- `ChannelFDefaultVideoFilter::ApplyFilter` converts VRAM to ARGB framebuffer

## Implementation Steps

### Phase 1: Accurate Port Write Timing (Medium Priority)

Current code applies VRAM writes instantly. Real hardware has:
- 1 CPU cycle for port write instruction
- VRAM access happens during specific phases of the video timing

**Implementation:**
1. Track current video timing (scanline/cycle within scanline)
2. Queue VRAM writes with timestamps
3. Apply writes at correct phase

### Phase 2: Overscan / Border Handling

Current code only renders the 128×64 visible area. Real hardware has:
- Horizontal: ~131 positions (128 visible + border)
- Vertical: 64 visible lines + VBlank period

**Implementation:**
1. Extend VRAM addressing to handle overscan writes (clamp or wrap)
2. Add border color from port 0
3. Configurable overscan visibility (off by default, useful for debugging)

### Phase 3: Frame Timing Synchronization

Real hardware generates video at:
- NTSC: ~60 fps, driven by crystal oscillator
- Each frame = 29829 CPU cycles (already correct)
- Scanlines: approximately 262 total (192 visible equivalent for NTSC TV)

The Channel F doesn't have scanline interrupts — the CPU just runs continuously and writes VRAM when it wants. Timing accuracy matters for:
- Games that race the beam (uncommon on Channel F but theoretically possible)
- Correct audio sync (audio toggles happen from same CPU)

### Phase 4: Background Color Register

Port 0 controls the background color. Current implementation may not properly handle:
- Background color changes mid-frame
- Interaction between background color and VRAM color 0

## Files Modified

- `Core/ChannelF/ChannelFMemoryManager.h/cpp` — VRAM write timing, overscan
- `Core/ChannelF/ChannelFDefaultVideoFilter.h/cpp` — Border color, overscan rendering
- `Core/ChannelF/ChannelFTypes.h` — Extended video state fields

## Testing Strategy

- Verify palette correctness: ARGB output matches expected colors (already tested)
- Verify VRAM bounds: X/Y clamping at overscan edges
- Verify background color changes
- Verify full-frame VRAM→ARGB conversion performance (already benchmarked)

## Risk Assessment

**Medium risk** — Phase 1 (write timing) could affect game rendering if not done carefully. Recommend implementing behind a flag first (`AccurateTiming = true/false`). Phases 2-4 are additive and low risk.
