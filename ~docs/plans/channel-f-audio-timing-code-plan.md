# Channel F Audio Timing — Code Plan (#979)

## Overview

The Channel F audio system is extremely simple: a single square wave generated by the CPU toggling an I/O port bit. There is no dedicated sound chip. This plan covers making the audio output accurate and artifact-free.

## Hardware Architecture

### Sound Generation

- **Single 1-bit output** — on or off
- CPU writes to port 5 bit 6 to toggle sound state
- Frequency determined entirely by CPU loop timing
- No hardware frequency counter, envelope, or modulation

### Current Implementation

In `ChannelFMemoryManager::WritePort()`:
- Port 5 writes update `_soundEnabled` flag
- Audio output is a simple square wave at CPU-driven frequency
- `ChannelFAudioState` tracks: `Period`, `Amplitude`, `Enabled`, `CycleCount`

### Real Hardware Behavior

1. CPU executes a tight loop: toggle port 5 → delay → toggle → delay
2. The delay loop length determines the pitch
3. Typical frequencies: 500 Hz – 4000 Hz (game-dependent)
4. Sound is synchronous with CPU — no buffering needed on real hardware

## Implementation Steps

### Phase 1: Cycle-Accurate Toggle Tracking

**Goal:** Record exact CPU cycle of each toggle for accurate waveform reconstruction.

1. On each port 5 write that changes bit 6, record the CPU cycle counter
2. Calculate half-period from consecutive toggles
3. Generate audio samples at the output sample rate by interpolating toggles

**Key formula:**

```
half_period_cycles = toggle_cycle[n] - toggle_cycle[n-1]
frequency = cpu_clock / (2 * half_period_cycles)
```

For NTSC: cpu_clock ≈ 1.7897725 MHz
For PAL: cpu_clock ≈ 1.7734474 MHz (see #1007)

### Phase 2: Anti-Aliasing / Band-Limiting

Raw square wave toggling produces aliasing at high sample rates. Options:
- **Simple:** Low-pass filter on output (current approach is likely fine)
- **Better:** BLEP (Band-Limited stEP) synthesis for clean edges
- **Best:** Precomputed BLEP table lookup (minimal CPU cost)

Recommendation: Simple low-pass filter is sufficient for Channel F's limited audio.

### Phase 3: Volume / Amplitude Control

Some Channel F games vary perceived volume by:
- Toggling at high frequency (ultrasonic) = perceived silence
- Toggling at audio frequency = perceived tone
- Not toggling = actual silence

Ensure the audio pipeline correctly handles:
- Very high toggle rates (>20 kHz) → output near-zero
- No toggles for extended periods → output zero

### Phase 4: PAL Clock Adjustment

When #1007 (console variants) is implemented:
- PAL variants use different CPU clock → different pitch for same code
- Audio timing calculations must use the active clock rate, not a hardcoded constant

## Risk Assessment

**Low risk** — Audio is a 1-bit toggle. Phase 1 is straightforward. BLEP synthesis (Phase 2) is optional polish. No risk to video or gameplay emulation.
